# Mudanças de 2026-06-26

## Resumo

Hoje foram trabalhadas regras novas para o mata-mata do Bolão V(devers), com foco em quatro frentes:

- Ocultar palpites dos participantes em jogos de mata-mata antes do jogo ficar ao vivo.
- Aplicar multiplicadores de pontuação por fase do mata-mata.
- Manter a pontuação usando o placar final oficial do jogo, incluindo prorrogação quando existir e sem somar pênaltis ao placar.
- Iniciar a estrutura de pontuação por chaveamento futuro, separada dos palpites normais de placar.

Também foi criada documentação visual na página de regras para explicar o funcionamento do mata-mata.

> Observação: o clone já tinha mudanças em outros arquivos antes destas alterações. As mudanças descritas aqui se referem ao que foi implementado hoje nos arquivos de palpites, pontuação, regras, serviços e migrations.

## Detalhamento

### 1. Palpites ocultos no mata-mata

Foi adicionada uma regra para jogos com `fase_id > 1`:

- Antes do jogo estar ao vivo, os palpites dos outros participantes não são exibidos publicamente.
- O participante autenticado continua vendo o próprio palpite.
- Quando `placar_status` passa para `live`, os palpites ficam visíveis para todos.
- Jogos da fase de grupos continuam com o comportamento anterior.

Locais alterados:

- `src/lib/server/bolao-service.ts`
- `src/lib/server/palpites-service.ts`
- `src/app/api/jogos/[id]/palpites/route.ts`

A regra também foi aplicada a agregados públicos, como dashboard e confiança da seleção, para evitar vazamento indireto de tendência de palpites antes do jogo.

### 2. Multiplicador de pontuação por fase

Foi criado um helper central de pontuação em `src/lib/scoring.ts`:

- `getPhasePointMultiplier(faseId)`
- `getPhaseAdjustedPoints(basePoints, faseId)`

Multiplicadores aplicados:

| Fase | `fase_id` | Multiplicador |
|---|---:|---:|
| 16 avos | 2 | 1.2 |
| Oitavas | 3 | 1.4 |
| Quartas | 4 | 1.6 |
| Semifinal | 5 | 1.8 |
| Final | 7 | 2.0 |

O cálculo usa `Math.floor`, então nunca há pontuação quebrada.

Exemplo em 16 avos:

- Chinelada: `10 * 1.2 = 12`
- Na trave: `7 * 1.2 = 8.4`, vira `8`
- Só o básico: `5 * 1.2 = 6`
- Deu sorte: `2 * 1.2 = 2.4`, vira `2`

### 3. Ranking e cálculo no banco

Foi criada a migration:

- `supabase/migrations/20260626010000_knockout_scoring_rules.sql`

Ela adiciona/atualiza:

- `public.multiplicador_pontos_fase(fase_id)`
- `public.calcular_pontos_placar(...)` com versão que recebe `fase_id`
- `public.recalcular_pontuacao_jogo(...)`
- `public.obter_ranking_ao_vivo()`

Assim, a pontuação exibida no app e a pontuação persistida no ranking usam a mesma regra.

### 4. Placar após prorrogação

O app continua usando `jogos.gols1` e `jogos.gols2` como placar oficial para pontuação.

Isso significa que:

- Se o jogo terminar em 120 minutos, `gols1/gols2` devem armazenar o placar após a prorrogação.
- Se houver disputa de pênaltis, os pênaltis não devem ser somados a `gols1/gols2`.
- Caso no futuro exista regra de vencedor nos pênaltis, ela deve ficar em campos separados.

Hoje o parser da TheSportsDB usa `intHomeScore` e `intAwayScore`; portanto, a regra depende de esses campos representarem o placar oficial do jogo, não a disputa de pênaltis.

### 5. UI da página do jogo

Em `src/app/jogos/[id]/page.tsx`, os cards de distribuição de pontuação passaram a mostrar os pontos já multiplicados conforme a fase.

Exemplo:

- Na final, Chinelada aparece como `+20 pts`.
- Nas quartas, Na trave aparece como `+11 pts`, pois `floor(7 * 1.6) = 11`.

### 6. Página de regras

Foi adicionado um bloco na página:

- `src/app/regras/page.tsx`

Esse bloco explica:

- Palpites ocultos no mata-mata.
- Prorrogação contando para o placar.
- Pênaltis fora do placar do bolão.
- Multiplicadores por fase.

### 7. Pontuação por chaveamento futuro

Foi iniciada a estrutura técnica para palpites de chaveamento futuro.

Arquivos adicionados:

- `supabase/migrations/20260626020000_chaveamento_palpites.sql`
- `src/lib/server/chaveamento-service.ts`
- `src/app/api/palpites/chaveamento/route.ts`

Arquivos alterados:

- `src/lib/queries.ts`
- `src/app/palpites/page.tsx`

Modelo criado:

- Tabela `public.palpites_chaveamento`
- Um registro por participante, fase e slot da chave
- Campos principais:
  - `user_id`
  - `fase_id`
  - `slot`
  - `time1`
  - `time2`
  - `vencedor`
  - `pontos`
  - `acertou`

Regra de pontuação planejada/estruturada:

- Cada confronto futuro acertado vale `5` pontos.
- A comparação ignora a ordem dos times.
- A pontuação é independente da pontuação dos jogos.
- Os pontos de chaveamento entram separados na API e também são somados ao ranking via função SQL.

Importante:

- A parte server/API/migration foi estruturada.
- A UI da aba de chaveamento em `src/app/palpites/page.tsx` começou a ser conectada, mas ainda precisa de acabamento para a chave interativa completa ficar pronta e validada visualmente.

## Como inserir hoje os jogos das 16 avos no banco

### Contexto do schema atual

A migration `20260612143000_add_rodada_and_16avos.sql` define:

- `fase_id = 2` para `16-avos`
- `fase_id = 3` para `Oitavas`
- `fase_id = 4` para `Quartas`
- `fase_id = 5` para `Semifinal`
- `fase_id = 6` para `Disputa de 3º`
- `fase_id = 7` para `Final`

A tabela usada é `public.jogos`.

Campos relevantes:

- `sportsdb_event_id`
- `fase_id`
- `time1`
- `time2`
- `data`
- `gols1`
- `gols2`
- `encerrado`
- `placar_status`
- `sportsdb_status`
- `rodada`

Para os 16 avos:

- usar `fase_id = 2`
- usar `rodada = null`
- usar `gols1 = null`
- usar `gols2 = null`
- usar `encerrado = false`
- usar `placar_status = 'upcoming'`
- preencher `sportsdb_event_id` quando houver o ID real da TheSportsDB

### Inserção recomendada

Depois que os classificados estiverem definidos, inserir os confrontos reais com `upsert` por `sportsdb_event_id`.

Exemplo:

```sql
insert into public.jogos (
  sportsdb_event_id,
  fase_id,
  time1,
  time2,
  data,
  gols1,
  gols2,
  encerrado,
  placar_status,
  sportsdb_status,
  rodada
)
values
  ('SPORTSDB_EVENT_M73', 2, 'Noruega', 'Espanha', '2026-06-28 13:00:00+00', null, null, false, 'upcoming', null, null),
  ('SPORTSDB_EVENT_M74', 2, 'Seleção A', 'Seleção B', '2026-06-28 16:00:00+00', null, null, false, 'upcoming', null, null)
on conflict (sportsdb_event_id) do update
set
  fase_id = excluded.fase_id,
  time1 = excluded.time1,
  time2 = excluded.time2,
  data = excluded.data,
  gols1 = excluded.gols1,
  gols2 = excluded.gols2,
  encerrado = excluded.encerrado,
  placar_status = excluded.placar_status,
  sportsdb_status = excluded.sportsdb_status,
  rodada = excluded.rodada;
```

### Cuidados ao inserir

- Não usar placeholders como `Vencedor M73` em `time1/time2` se a página de palpites de chaveamento depender de confrontos reais.
- Só liberar a aba de chaveamento quando os confrontos iniciais do mata-mata estiverem definidos.
- Preencher `sportsdb_event_id` real para permitir sync automático de placar.
- Usar datas em `timestamptz`.
- Se os horários oficiais ainda não estiverem confirmados, inserir apenas quando houver horário confiável ou atualizar depois via `on conflict`.
- Para placar com prorrogação, ao finalizar o jogo, `gols1/gols2` devem representar o placar ao fim dos 120 minutos, sem pênaltis.
